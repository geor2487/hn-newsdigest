## 1. Qwen 3.5をローカルで動かす方法
**原題**: How to run Qwen 3.5 locally
**URL**: [https://unsloth.ai/docs/models/qwen3.5](https://unsloth.ai/docs/models/qwen3.5)
**HN**: [https://news.ycombinator.com/item?id=47292522](https://news.ycombinator.com/item?id=47292522)
**スコア**: 370 / **コメント**: 113件 / **合計**: 483

<span style="color: orange;">**ざっくり言うと:**</span>
Alibaba（アリババ）が出した新しいAIモデルファミリー「Qwen 3.5」を、自分のパソコンで動かすためのガイド。Qwen 3.5は0.8Bから397Bまで幅広いサイズがあり、22GBのメモリを持つMacなら35Bや27Bモデルが動かせる。256Kトークンのコンテキスト長（一度に処理できるテキスト量）、201言語対応、思考モード/非思考モードの切り替えなど、かなり高機能。Unslothというツールが独自の量子化（GGUF（モデルを圧縮して軽くするフォーマット））を提供していて、4bitでも重要なレイヤーを8bitや16bitに自動的に引き上げる「Dynamic 2.0」方式を採用している。

<span style="color: orange;">**もう少し詳しく:**</span>
Qwen 3.5はMoE（Mixture of Experts（全パラメータのうち一部だけを使って推論する効率的なアーキテクチャ））を採用しているモデルが含まれている。例えば35B-A3Bは全体で350億パラメータだが、実際に使われるのは30億パラメータだけなので、見た目のサイズより遥かに軽い。Unslothはこれらのモデルを一般的なハードウェアで動かせるようにGGUFフォーマットで量子化を行っている。量子化とは、モデルの精度を多少犠牲にしてメモリ使用量を大幅に減らす手法。最新のアップデートではimatrix（重要度行列）データを改善し、チャット・コーディング・長文コンテキスト・ツール呼び出しの性能が向上した。antirez氏（Redisの作者）のベンチマークによると、think（思考）モードではDeepSeekの83%程度の性能だが、no-thinkモードでは53%程度。ただしno-thinkモードは圧倒的に速いため、用途によってはこちらの方が実用的。消費者向けGPUでも100トークン/秒の速度が出るとの報告もあり、オンラインLLMサービスを上回る速度。

<span style="color: orange;">**なぜ注目？:**</span>
ローカルLLM（大規模言語モデル）の民主化が急速に進んでいる象徴的な記事。22GBのMacでそこそこ賢いAIモデルが動くというのは、つい1年前では考えられなかった。プライバシーが気になる用途やオフライン環境での利用、APIコストの削減など、ローカル実行のメリットは大きい。特にMoEアーキテクチャの普及により「パラメータ数は大きいが実行は軽い」モデルが増えており、今後もこのトレンドは加速するだろう。

## 4. PyPyがメンテナンスされなくなっている問題
**原題**: Warn about PyPy being unmaintained
**URL**: [https://github.com/astral-sh/uv/pull/17643](https://github.com/astral-sh/uv/pull/17643)
**HN**: [https://news.ycombinator.com/item?id=47293415](https://news.ycombinator.com/item?id=47293415)
**スコア**: 276 / **コメント**: 145件 / **合計**: 421

<span style="color: orange;">**ざっくり言うと:**</span>
Pythonのパッケージマネージャ「uv」に、PyPy（Pythonの高速な代替実装）が活発に開発されていないことを警告するPR（Pull Request（コードの変更提案））が出された。PyPyはCPython（通常のPython）よりも最大5倍以上高速に動作するJIT（Just-In-Time（実行時にコードをコンパイルして高速化する技術））コンパイラを搭載した実装だが、コア開発者が不足しておりPython 3.12対応すら完了していない。numpyもPyPyのサポートを段階的に終了しつつある。

<span style="color: orange;">**もう少し詳しく:**</span>
PyPyのコア開発者であるcfbolztereick氏がHNコメントで直接説明している。「PyPyはメンテナンスされていないわけではない。バグ修正やJITの改善は行っている。ただし、CPythonの新バージョンに追従する人手が足りない」とのこと。PyPyは技術的には素晴らしい成果で、Microsoftの「Faster CPython」チームが4年かけて1.5倍の高速化しか達成できなかったのに対し、PyPyは数十年にわたり5倍以上の高速化を実現してきた。しかし資金と人材の問題でCPythonの進化速度についていけなくなっている。PRのタイトルが当初「unmaintained（メンテナンスされていない）」だったのが「no longer under active development（活発な開発が行われていない）」に変更されるなど、表現を巡る議論も起きた。

<span style="color: orange;">**なぜ注目？:**</span>
オープンソースプロジェクトの持続可能性という根本的な問題を浮き彫りにしている。技術的に優れたプロジェクトでも、資金と人材がなければ維持できない。PyPyを使っているプロジェクトは移行計画を考える必要がある。また、PyPI（パッケージインデックス）とPyPy（Python実装）を混同する人が多いので注意。

## 5. CasNum — コンパスと定規で任意精度演算
**原題**: CasNum
**URL**: [https://github.com/0x0mer/CasNum](https://github.com/0x0mer/CasNum)
**HN**: [https://news.ycombinator.com/item?id=47291292](https://news.ycombinator.com/item?id=47291292)
**スコア**: 318 / **コメント**: 38件 / **合計**: 356

<span style="color: orange;">**ざっくり言うと:**</span>
「コンパスと定規」という古代ギリシャの作図法だけで任意精度の算術演算を実装したライブラリ。さらに、ゲームボーイエミュレータのALU（Arithmetic Logic Unit（演算論理装置: CPUの計算を担当する部分））の全命令を幾何学的作図で実装するという壮大な（そしてまったく実用的でない）プロジェクト。ユーモアあふれるドキュメントも話題に。

<span style="color: orange;">**もう少し詳しく:**</span>
コンパスと定規による作図とは、ユークリッド幾何学の古典的な手法で、円を描くコンパスと直線を引く定規だけで図形を構成する方法。CasNumはこの手法を使って足し算、引き算、掛け算、割り算などの基本演算を実装している。例えば、2つの線分の長さの比を使って割り算を表現するといった具合。さらに驚くべきことに、改造版ゲームボーイエミュレータを組み込んでおり、ALUのオペコード（CPUの個々の命令）をすべて幾何学的作図で置き換えている。FAQの「速いのか？」に対する回答「ユークリッドを手で写すより速い」、「Doomは動くか？」に対する「これは数字であって、何かを'動かす'ものではない」といったユーモアも人気の理由。モール＝マスケローニの定理（コンパスだけで定規と同じ作図ができるという定理）を使えばさらに簡素化できるという指摘も。

<span style="color: orange;">**なぜ注目？:**</span>
実用性はゼロだが、計算の本質を古代の数学で表現するというアイデアが面白い。コンピュータサイエンスの根底にある「計算とは何か」という問いに、全く違う角度から光を当てている。プログラミングのユーモアと数学の美しさが融合した好例。
