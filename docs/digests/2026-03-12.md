## 1. AI生成コメントを投稿するな — HNは人間同士の会話の場だ

## 3. Temporal: JavaScriptの「時間」を修正する9年間の旅
**原題**: Temporal: The 9-year journey to fix time in JavaScript
**URL**: https://bloomberg.github.io/js-blog/post/temporal/
**HN**: https://news.ycombinator.com/item?id=47336989
**スコア**: 587 / **コメント**: 197件 / **合計**: 784

**ざっくり言うと:**
JavaScriptの日付・時刻処理を根本的に改善する新しいAPI「Temporal」が、9年の開発期間を経てついにブラウザに搭載され始めた。従来の`Date`オブジェクトは設計上の問題が多く、開発者を長年悩ませてきた。Bloombergのエンジニアが中心となって進めたこの提案の道のりが詳しく語られている。

**もう少し詳しく:**
JavaScriptの`Date`オブジェクトは1995年にJavaから移植されたもので、タイムゾーン処理が不完全、ミュータブル（変更可能: 一度作ったオブジェクトの中身を後から書き換えられてしまう設計）、月が0始まりなど、多くの落とし穴がある。そのため、moment.jsやdate-fnsといった外部ライブラリが事実上の標準として使われてきた。Temporalは TC39（JavaScriptの標準化委員会）のプロセスを通じて、2017年のStage 1から2026年のStage 4まで段階的に進められた。「瞬間（Instant）」と「カレンダー上の日時」を明確に区別するなど、時間の本質的な複雑さに正面から向き合った設計になっている。Firefoxではボランティアの André Bargull が単独で実装を完了させた。

**なぜ注目？:**
Web開発者なら誰もが苦しんだ`Date`の問題がついに解決に向かう。ブラウザネイティブでタイムゾーン対応の日時処理ができるようになれば、moment.jsのような大きなライブラリを読み込む必要がなくなり、Webアプリのバンドルサイズ削減にも貢献する。9年という期間は、Web標準化の難しさと重要性を物語っている。

## 4. WebAssemblyをWeb上の「一級市民」にする
**原題**: Making WebAssembly a first-class language on the Web
**URL**: https://hacks.mozilla.org/2026/02/making-webassembly-a-first-class-language-on-the-web/
**HN**: https://news.ycombinator.com/item?id=47331811
**スコア**: 476 / **コメント**: 166件 / **合計**: 642

**ざっくり言うと:**
Mozillaのエンジニアが、WebAssembly（Wasm: ブラウザ上で高速に動くバイナリ形式のプログラム）が現在「二級市民」として扱われている問題を指摘し、解決策としてWebAssembly Componentsを提案している。2017年のリリース以来、言語機能は充実してきたが、Web APIとの統合が不十分なままだ。

**もう少し詳しく:**
Wasmは言語としてはSIMD（Single Instruction Multiple Data: 1つの命令で複数のデータを同時処理する技術）、GC（ガベージコレクション: 不要になったメモリを自動で解放する仕組み）、例外処理など多くの機能が追加されてきた。しかし、DOM操作やWeb APIへのアクセスには常にJavaScriptを経由する必要があり、開発者体験が悪い。この「Wasmの崖」が採用を阻んでいる。WebAssembly Componentsは、WasmモジュールがWeb APIに直接アクセスできるようにする仕組みで、言語固有のグルーコード（異なるシステムを繋ぐための仲介コード）を不要にすることを目指している。現状ではWasmを使うのは大企業が中心で、個人開発者やスタートアップにとっては導入コストが高すぎる。

**なぜ注目？:**
Wasmが真にWebの一級市民になれば、Rust・Go・C++などあらゆる言語でWebアプリを直接書ける世界が近づく。JavaScriptの独占が終わる可能性を秘めた、Web の未来に関わる重要な議論だ。

## 6. BitNet: CPUで動く1000億パラメータの1ビットモデル
**原題**: BitNet: 100B Param 1-Bit model for local CPUs
**URL**: https://github.com/microsoft/BitNet
**HN**: https://news.ycombinator.com/item?id=47334694
**スコア**: 323 / **コメント**: 160件 / **合計**: 483

**ざっくり言うと:**
MicrosoftがBitNet b1.58の推論フレームワーク「bitnet.cpp」を公開した。通常のLLM（大規模言語モデル）が16ビットや32ビットの浮動小数点数を使うのに対し、わずか1.58ビット（-1、0、1の3値）でモデルの重みを表現する。1000億パラメータのモデルでも単一CPUで人間の読書速度（5〜7トークン/秒）で動作する。

**もう少し詳しく:**
従来のLLMは膨大なメモリと計算資源を必要とし、ローカル実行はハイエンドGPUが前提だった。BitNetは重みを3値（-1、0、1）に量子化（精度を下げてデータサイズを圧縮する手法）することで、メモリ使用量と消費電力を劇的に削減する。ARM CPUで1.37〜5.07倍の高速化、55〜70%の省エネを達成。x86 CPUでは2.37〜6.17倍の高速化と71〜82%の省エネ。ただし、同等の性能を得るにはFP16（16ビット浮動小数点）モデルの4〜5倍のパラメータ数が必要とされる。現時点ではGPU推論もサポートされ、NPU（Neural Processing Unit: AI処理専用チップ）対応も予定されている。

**なぜ注目？:**
LLMをクラウドに頼らずローカルで動かせるようになる可能性を広げる技術。プライバシーやコスト、オフライン利用の面で大きなインパクトがある。ただし「1ビット」というタイトルはやや誤解を招きやすく、実際は「1.58ビット（3値）」であり、100Bモデルが訓練済みで公開されているわけではなく推論フレームワークが公開された段階という点に注意。
