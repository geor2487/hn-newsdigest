## 1. トニー・ホーア氏が逝去
**原題**: Tony Hoare has died
**URL**: https://blog.computationalcomplexity.org/2026/03/tony-hoare-1934-2026.html
**HN**: https://news.ycombinator.com/item?id=47324054
**スコア**: 1458 / **コメント**: 197件 / **合計**: 1655

<span style="color: orange;">**ざっくり言うと:**</span>
コンピュータ科学の巨人、トニー・ホーア（Sir Tony Hoare）氏が3月5日に92歳で亡くなった。ホーア氏は、プログラマなら誰でも学ぶ「クイックソート（Quicksort: データを高速に並び替えるアルゴリズム）」の発明者として最も有名だが、それ以外にも「ホーア論理（Hoare Logic: プログラムが正しく動くことを数学的に証明する手法）」や、プログラミング言語ALGOLの設計、CSP（Communicating Sequential Processes: 並行処理を安全に行うための理論的フレームワーク）など、計り知れない貢献を残した。1980年にはチューリング賞（Turing Award: コンピュータ科学のノーベル賞とも呼ばれる最高の栄誉）を受賞している。

<span style="color: orange;">**もう少し詳しく:**</span>
ホーア氏の業績はコンピュータ科学の根幹に深く刻まれている。クイックソートは1960年に発表されたが、60年以上経った今でも実用的なソートアルゴリズムとして広く使われている。ホーア論理は「このプログラムは仕様通りに動く」ということを数学的に証明するための体系で、ソフトウェアの信頼性を高める基盤となった。また、CSPという並行処理の理論は、Go言語のチャネル（goroutine間でデータをやり取りする仕組み）など、現代のプログラミング言語の設計にも直接的な影響を与えている。彼はオックスフォード大学の教授として長年研究を率い、その後マイクロソフトリサーチでも活躍した。興味深いことに、ホーア氏は「null参照（何も指していないポインタ）」を発明したことを「10億ドルの間違い」と自ら呼んだことでも知られている。この率直さと謙虚さが、彼の人柄をよく表している。個人的な回想記事では、温かく知的好奇心に満ちた人物として描かれている。

<span style="color: orange;">**なぜ注目？:**</span>
ホーア氏の仕事は、普段意識されないだけで、ほぼすべてのプログラマーの日常に影響を与えている。ソートアルゴリズムを学んだ人なら必ず出会うクイックソート、Go言語やErlangの並行処理モデルの理論的基盤となったCSP — これらはすべてホーア氏の発明だ。コンピュータ科学の歴史の中でも特に大きな足跡を残した人物の死去であり、HNコミュニティでも多くの追悼と感謝のコメントが寄せられている。

## 2. 子ども向け年齢確認ツールが、大人を監視している
**原題**: Online age-verification tools for child safety are surveilling adults
**URL**: https://www.cnbc.com/2026/03/08/social-media-child-safety-internet-ai-surveillance.html
**HN**: https://news.ycombinator.com/item?id=47322635
**スコア**: 533 / **コメント**: 301件 / **合計**: 834

<span style="color: orange;">**ざっくり言うと:**</span>
アメリカの約半数の州が、未成年をオンラインの有害コンテンツから守るために「年齢確認」を義務化する法律を制定・推進している。しかしこの仕組みは、子どもだけでなく全てのユーザーに身分証明や顔スキャンを要求するため、結果として大人のプライバシーが大規模に侵害される事態になっている。「子どもを守るため」という大義名分の裏で、自由でオープンなインターネットが脅かされているとの批判が強まっている。

<span style="color: orange;">**もう少し詳しく:**</span>
これらの法律は、アダルトサイト、オンラインゲーム、SNSなどに対して未成年のアクセスをブロックすることを求めている。問題は「年齢確認 = 身元確認」になってしまう点だ。年齢だけを匿名で確認する技術的な仕組みが確立されていないため、運転免許証のアップロードや顔認証など、個人を特定できる情報の提出が必要になる。イギリスのOnline Safety Actでは当初、コンビニで匿名のIDコードを現金で買える仕組みが提案されていたが、実現しなかった。つまり「あなたが18歳以上かどうか」を確認するだけのはずが、「あなたが誰で、何を見ているか」まで把握される仕組みになっている。FTC（連邦取引委員会: アメリカの消費者保護機関）は、収集した情報の利用制限を求めているが、一度収集されたデータが漏洩やミスで流出するリスクは常にある。

<span style="color: orange;">**なぜ注目？:**</span>
「子どもの安全」と「プライバシー」のバランスは、テクノロジー業界で最も難しい問題の一つだ。多くのHNコメントが指摘するように、この法律の本当の目的は「大人の匿名性を奪うこと」ではないかという懸念がある。特にオンラインでの発言を取り締まる動きが強まっている国では、年齢確認が事実上の身元確認・監視ツールになりうる。初学者にとっても、自分のオンライン活動がどこまで追跡されるかを考える重要なきっかけになる話題だ。

## 4. Redox OSが「出所証明書」ポリシーと厳格なLLM禁止ポリシーを採用
**原題**: Redox OS has adopted a Certificate of Origin policy and a strict no-LLM policy
**URL**: https://gitlab.redox-os.org/redox-os/redox/-/blob/master/CONTRIBUTING.md
**HN**: https://news.ycombinator.com/item?id=47320661
**スコア**: 372 / **コメント**: 370件 / **合計**: 742

<span style="color: orange;">**ざっくり言うと:**</span>
Rust言語で書かれたOS「Redox OS」が、コントリビューションガイドラインを更新し、2つの重要なポリシーを導入した。1つは「Certificate of Origin（出所証明書）」— 貢献者が自分のコードの著作権者であることを宣言する仕組み。もう1つは「LLM（Large Language Model: ChatGPTなどの大規模言語モデル）で生成されたコードは一切受け付けない」という厳格なポリシーだ。LLM生成と明示されたIssueやMR（Merge Request: コード変更の提案）は即座にクローズされ、このポリシーを回避しようとした場合はプロジェクトから永久追放される。

<span style="color: orange;">**もう少し詳しく:**</span>
Certificate of Originは、Linux Foundationの「Developer Certificate of Origin（DCO）」にならった仕組みで、コントリビューターが3つの条件に同意することを求める。「自分が著作権者であること」「MITライセンスで提出すること」「変更内容を十分理解していること」の3点だ。LLM禁止ポリシーの背景には、OSSメンテナーの負担増がある。LLMで生成されたコードは一見それらしく見えるが、微妙なバグを含んでいたり、プロジェクトの文脈を理解していない変更だったりすることが多い。メンテナーはこれらを一つ一つレビューしなければならず、結果として「質の低い大量のPR（Pull Request: コード変更の提案）」に時間を奪われてしまう。NetBSD、GIMP、Zig、QEMUなど、他のメジャープロジェクトも同様の禁止ポリシーを採用している。

<span style="color: orange;">**なぜ注目？:**</span>
OSSコミュニティでのAI生成コードの扱いは、今最もホットな議論の一つだ。「AIを使えば誰でもコントリビュートできる」という民主化の側面と、「低品質なコードが大量に押し寄せてメンテナーが疲弊する」という現実のトレードオフがある。この記事の直後に掲載されたDebianの記事（本ダイジェスト8番）と合わせて読むと、OSSプロジェクトごとにAIへの対応が全く異なることがわかり、興味深い。

## 用語集

| 用語 | 解説 |
|------|------|
| **クイックソート（Quicksort）** | データを高速に並び替えるアルゴリズム。1960年にトニー・ホーアが発明し、現在も広く使われている |
| **チューリング賞（Turing Award）** | コンピュータ科学の最高の栄誉。「コンピュータ科学のノーベル賞」とも呼ばれる |
| **LLM（Large Language Model）** | ChatGPTなどの大規模言語モデル。大量のテキストデータから学習し、文章の生成や質問への回答ができる |
| **ワールドモデル（World Model）** | AIが物理世界の仕組み（重力、物体の動きなど）を理解・予測するためのモデル |
| **ファインチューニング（Fine-tuning）** | 既に学習済みのAIモデルに追加のデータで学習させ、特定のタスクの性能を向上させること |
| **エージェント（AI Agent）** | 人間の指示に基づいて自律的にタスクを実行するAIプログラム |
| **テストシアター（Test Theatre）** | テストが通っているように見えるが、実際には何も検証していない見せかけのテスト |
| **PR / MR（Pull Request / Merge Request）** | コードの変更を提案し、レビューを経てメインのコードに統合してもらうための仕組み |
| **CSP（Communicating Sequential Processes）** | 並行処理を安全に行うための理論。Go言語のチャネルなどに影響を与えた |
| **GR（General Resolution）** | Debianプロジェクトで全メンバーの投票により重要な方針を決定する手続き |
